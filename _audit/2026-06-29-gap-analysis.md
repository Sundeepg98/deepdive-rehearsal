## 1. Headline — the missing-ness has two shapes, and they collapse onto two keystones

**Shape 1 — BREADTH is the dominant gap: a polished framework with content for exactly one of eight promised topics.** The hard engineering is done and is *almost* topic-parameterizable — 9 surfaces, 7 overlays, the HashRouter, the timed mock-run, drill self-scoring, session-progress. But `index.html:22` hardcodes `Topic 1 of 8 · ingestion layer`, there is **no topic switcher anywhere**, and the `.topic-item`/`topic-list` CSS sits **orphaned**. The other seven topics are not invented — they are already **named and bridge-authored in-product**: `SYS_PIVOTS` writes the interviewer pivots into **Signing (17), Authz (18), AWS-hardening (19), Notifications (20), EAV (21), Desired-state (22), IaC (23)** ("IaC drift detection is to infrastructure what the reconciler is to data"). The pivot prose advertises destinations that don't exist. This is the single highest-value missing thing — and, honestly, the **largest** lift (content-XL).

**Shape 2 — DEPTH-OF-MEMORY: even within the one topic, the entire retention-over-time half is absent — and the product says so itself.** The progress code is a **lossy aggregate snapshot**: `encodeSession()` stores only counts (`dGot/dShk/dDone/dTot`, wb tallies, mock score, mixed totals) and throws away *which* signals were weak — even though `this.revisit` already holds those exact identities in memory, keyed by `_allCards` index. The game-plan overlay literally promises behavior the engine can't keep ("find your weak signals and re-drill those"; "paste yesterday's to watch the trend climb"). The code is a **scoreboard, not a save-file**, so every session re-grades from scratch and no spaced re-drilling is possible.

**The synthesis insight — five dimensions, two load-bearing refactors.** When you lay all five ideation passes side by side, almost everything high-value rides on exactly two keystones:

- **Keystone A — the per-topic DATA contract.** Normalize all 9 surfaces to one declarative `TOPIC={id,label,locator,surfaces:{…}}` object, then add a registry + `#/<topic>/<surface>` router param + switcher. Five surfaces (walk, drill, mock, model, wb) already *are* clean data arrays; the keystone work is converting the four HTML-baked surfaces — **trade-offs (`TRADE_HTML`), numbers (`NUM_HTML`+`_calc`), red-flags (`RF_HTML`), opener (`OP_HTML`)** — into data arrays their components render. `getDecisions()` already round-trips trade-offs DOM→data, proving the schema *wants* to be uniform. **This is the precondition for ALL of curriculum** (breadth *and* depth): until it lands, every new topic means hand-authoring HTML four times over, and trade-offs/numbers/red-flags can never be drilled or expanded.

- **Keystone B — the CPR2 per-signal progress code.** Put **stable per-signal IDs + per-item mastery/outcome state** into the code (extend `CPR1.`→`CPR2.`, backward-compatible because `decodeSession`'s trailing group is already optional). This one upgrade is requested by **two dimensions from two directions**: learning-science wants it as a Leitner box/interval vector (spaced repetition), persistence wants it as a per-signal outcome ledger (weakness heatmap + save-game resume). **They are the same code.** Build it once and both halves unlock.

And one **honesty substrate** sits upstream of Keystone B: today *every* self-score is a subjective gestalt ("did that feel clean?"), never checked against the load-bearing points the answer actually requires. Persisting vibes per-signal is garbage-in. So the **must-hit-points checklist** (rehearsal dimension) is what makes CPR2's per-signal data *worth* storing — it must land alongside or just before it.

Everything else — the a11y spine, preferences, personas, time-budgets, week-plans, exports — is comparatively well-supported leverage *on top of* these. Two of those (the a11y spine and preferences) are **orthogonal** to both keystones (pure markup/ARIA and CSS custom-props), which makes them the ideal parallel track that never contends for the same files.

## 4. Build order (dependency-driven)

The operator's standard flow applies: **lay the shared contract first → parallel BUILD in worktree-isolated agents → serial VERIFIED merge, one at a time, GATE-gated.** The dependency graph collapses to three phases plus an orthogonal track.

**Phase 0 — Keystones + honesty substrate (do first; these are the shared contracts).** They touch different subsystems (content/render vs. the code vs. mock/drill data), so they can be worktree-isolated parallel builds merged one-at-a-time through THE GATE — with one soft coupling: define CPR2's signal-ID scheme *before* the checklist persists outcomes against it.
1. **Must-hit-points checklist** (Rehearsal, M) — grounds every self-score; mostly independent; ship first so CPR2 stores honest data, not vibes.
2. **Topic DATA contract + registry/switcher** (Keystone A, L) — data-ize the 4 HTML-baked surfaces + registry + router param + switcher. No new topics yet; just the pipeline re-expressed as data. Makes `Topic 1 of 8` honest.
3. **CPR2 per-signal mastery code** (Keystone B, L) — stable IDs + mastery/outcome state, backward-compatible. **Design topic-namespacing in from day one** so it survives Keystone A's multiplication of topics.

**Phase 1 — Consumers that ride directly on the keystones (parallelizable).**
- *On Keystone B:* History-aware weighting (S) → Due-today queue (M) → Graded recall (M) → mastery map / leech detection / taper scheduler; and on the persistence side: weakness heatmap, **Save-game resume**, accreting study-log, PDF trend export.
- *On Keystone A:* **Topic-authoring template + GATE content lint** (S) — build this BEFORE mass authoring; generalized NALSD engine; trade-offs drillable bank; per-tier depth floors + EXTEND filter.

**Phase 2 — The big content bet + richer realism (the long pole).**
- **Author the 7 sibling topics** (XL) — gated behind Keystone A + template + lint. One warm authoring agent per topic, worktree-isolated, **serial verified-merge**. Price one topic end-to-end first.
- Concept index + curriculum map — only pay off at ≥2 topics, so they land *with* authoring.
- Rubric scorecard, per-beat time budgets, interviewer personas, adaptive week-plan — realism/scheduling depth.

## 6. Honest caveats

- **The 7 topics are content-XL, not an engineering feature.** Keystone A + the template/lint make them parallelizable and machine-gated, but the authoring itself is the long pole of the whole roadmap. Price one topic end-to-end before committing to all seven, and settle the per-topic-files-vs-monolith build decision when the registry lands.
- **CPR2 is worth building only with the must-hit-points checklist upstream of it.** Persisting per-signal *vibes* is garbage-in; the checklist is what makes the stored per-signal outcomes meaningful. Sequence the honesty substrate first.
- **Several "HIGH/S" learning-science wins are gated on CPR2.** History-aware weighting degrades gracefully to today's random behavior without a loaded code, so it *ships* cheap — but its payoff only materializes once the mastery vector exists.
- **The a11y spine is the one track that should NOT wait.** It is orthogonal to both keystones (markup/ARIA), and it is the difference between the core graded loop being operable or unusable for the keyboard/SR-only candidate who most benefits from an offline trainer. Run it in parallel from day one.
