## THE INVARIANT — restore-never-regrades (proven at the mechanism level)

Grades enter the persisted record ONLY via `judge()` → `drillgraded` → `Progress.snapshot()`, which
MERGES by content-id (progress.js:200-208: starts from the prev record's card map, updates only this
session's graded probes by `g[k].id`; so grading 5..end updates those ids in place and never touches
0..4). The cursor restore only sets `this.di` and re-renders (`drawCard` shows the probe's question);
it never calls `judge()`, so it never snapshots. Restore is structurally incapable of re-grading.

## Deviations from §60 (all deliberate, flagged for w0-verifier)

**D1 — drill debrief denominator (this-run, not full-bank), on the W1-verified drill verdict surface.**
The WIP's `pct = got / results.length` (probes ANSWERED THIS RUN) replaces `got / cards.length`.
Accepted per team-lead's Beat-1 ruling (it kills the resume-penalty with a smaller blast radius than
compute-from-record, and full-run equivalence holds STRUCTURALLY: full run ⇒ results.length ==
cards.length ⇒ identical old math — not dependent on record-merge semantics). Conditions met: (i) the
resumed-run copy is un-misreadable ("of N answered this run", never implying full-bank mastery;
cumulative truth stays in Progress/session panel); (ii) the structural-equivalence case is IN the gate.

**D2 — completion-reset.** A study walk that reaches the debrief (di >= cards.length) resets the
cursor to 0, so re-entering a FINISHED drill lands on the study list at probe 1 — the pre-cursor
behavior progress_merge / card_identity rely on — instead of the last graded probe. (A partial walk
persists its di.) This surfaced as a progress_merge failure and is the honest fix.

**D3 — "drill PROBE" (single index, study mode only), not the brief's looser "mode+di".** The
spec/§1c says "drill probe" (sibling to "walk step"). Only the canonical full-bank study walk is a
true bank cursor; restoring MOCK mode would auto-start its 22-min timer (an unrequested side effect,
anti-goal), and QUICK reshuffles 5 random cards. So the cursor persists+restores the study-walk probe
only. Documented deviation from the mission-brief phrasing (document-wins).

**D4 — boot-safe flowRec (a latent W1 bug my durability fix exposed).** The pagehide/visibility
cursor flush made `pos.walk=lastStep` persist across a reload; on the next boot the walk's last-step
morph calls `flowRec()` inside `connectedCallback`, before a sibling pane is upgraded or
mixed-fire/mock-run have run their top-level init. `sessStats`/`sessLiveActivity` threw
(`drillEl().getStats is not a function`, then `mixLog.length`). Fixed by guarding getStats against an
UN-UPGRADED element (not just null) and mixLog/mock globals against being UN-INITIALIZED — falls back
to the canonical record, identical once loaded. Root-caused from the stack; caught by progress_merge +
flow_handoff going red. This is a robustness fix to a W1 function, flagged as such.

**D5 — home CTA keeps RESUME semantics; NOT §60's "home CTA renders NextUp large".** I surfaced the
tension (flowRec is per-current-topic and suppressed on #home; the conversion needs a record-only
`sessStatsFor(id)` adapter + changes the daily-loop Enter target). Team-lead ruled it out of scope.
**Rationale, verbatim from `_TEAM_LEAD_BEAT2_RULING.md`:**
