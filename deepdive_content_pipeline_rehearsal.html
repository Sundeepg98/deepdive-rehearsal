Cram sheet · Content Pipeline
The one-linerEvent-driven ingestion: S3 → Lambda → route by type → streaming handler (hash + store + record) → reconciler for partial failures.
The spine — what you draw
  1. Entry: processUpload(key, bucket), fired by S3 ObjectCreated.
  2. Route: extname → O(1) strategy map, never a switch.
  3. Unknown ext → skip; a match → the format handler.
  4. Handler: readStream → PassThrough → [hash | s3.upload] → Promise.all — one read.
  5. Export: cursor(100) → csv → res + backpressure — constant memory.
  6. Import ladder: REUSE / REMAP / REGEN / INSERT + oldId→newId FK remap.
  7. Dual-write: track S3 keys + compensating-delete on rollback.
  8. Reconciler sweeps orphans — guarded by grace window / PENDING marker.
  9. Replay-safe: processed-marker (conditional put) — idempotent.
Decisions & switch conditions
Lambda per object→SQS the moment you need retries / DLQ / ordering.
SQLite / Postgres over JSON→need real FKs + 1:1 BLOBs.
Hash during upload (fork)→2× read or buffering won't survive 500 MB.
Inline sync→async ECS / MediaConvert once transcoding is heavy.
Ceilings — the numbers
Lambda concurrency > 1,000 → buffer through SQS.
DB connections > pool → RDS Proxy.
S3 PUT / prefix rate → spread key prefixes.
10M files/day ≈ 116/s avg · ~1,157/s peak · ~$50/day in PUTs.
Traps → the fix
load all rows then write CSV→cursor + backpressure
hash after store→PassThrough fork
switch() routing→O(1) dispatch map
two writes = consistent→compensate + reconciler
exactly-once delivery→idempotent effects
just retry→DLQ + idempotency
reconciler deletes orphans→grace / PENDING guard
Senior tells — say these
  • Name the switch condition, don't defend one choice.
  • Memory is constant because of backpressure.
  • Volunteer the reconciler as the real backstop.
  • Reframe: exactly-once delivery is impossible → idempotent effects.
  • Name the resource that gives out first.
Harder angles — curveball-ready
Operate — four signals: queue depth · latency p99 · error rate · orphan count (the correctness canary); alert on SLO burn, not single blips.
Security — the strategy map is the allowlist; validate by content not extension; sandbox the processor; a per-job timeout contains a poison input.
Cost — at firehose scale per-object Lambda is the expensive choice → batch through SQS; lifecycle cold objects to Glacier.
Ordering — S3 events aren’t ordered → SQS FIFO per key or a sequence number; first challenge whether you need global order.
Backfill — reprocess through the same pipeline, rate-limited, with reserved concurrency so it can’t starve live traffic; safe because idempotent.
Cut scope — MVP is S3→Lambda→handler→row; defer DLQ / multipart / reconciler; never cut idempotency or the strategy-map seam.
If they say “quickly” — the 30 seconds
Operators push to S3 → the processor routes by type through an O(1) map → each handler streams once, forking the read to hash + upload, so memory stays flat. Two stores, no shared txn → track keys, compensate, reconciler backstop. At-least-once → a processed-marker makes replays no-op.

Scope it first · the questions before the design
The first signalBefore you draw anything, scope it. The first thing the interviewer reads is whether you solution blindly or pin down the problem. Ask the questions whose answers would change your design — not cosmetic ones. Here are the ones that fork this architecture, and what each answer flips.
What we’re ingesting
File types & formats? — a few known types means a static strategy map; arbitrary user uploads means pluggable handlers and a hard ‘unknown type’ policy.
Size — KB configs or GB media? — the single biggest fork. Small → simple buffered handlers; large → streaming, multipart, transcode off the hot path.
Processing — light or heavy? — hash and parse are CPU-light, fine inline; transcode or ML is heavy → async to a worker pool, never inside the Lambda.
Scale
Objects per day, and peak-to-average? — this sets the ceilings; it’s exactly what the Numbers tab plugs in. Peak/s × processing time is your concurrency.
Bursty or steady? — bursty means you need a queue to absorb spikes; steady may not.
Who consumes the output
Sync API, or poll / subscribe? — flips whether you owe a status API or webhooks, or just a record.
Latency — minutes fine, or someone waiting? — sets the whole sync vs async posture.
Correctness & durability — the senior questions
Same upload processed twice — catastrophic or just wasteful? — sets the idempotency bar.
Can we ever drop an upload? — at-least-once and a durable queue, or best-effort.
Retention, deletion, compliance? — decides whether you need provenance and tombstones from day one.
Bound it — what you’re NOT doing
Designing the upload path, or does the object already land in storage? — bounds where the design even starts.
Then say what’s out of scope — auth, the client UI, billing. Naming non-goals stops you sprawling and shows you scope on purpose.
Cosmetic vs forking — hear the difference
‘What language should I use?’

The tellJuniors ask nothing and start drawing, or ask cosmetic questions. Seniors ask the 3–4 whose answers fork the architecture — size, sync vs async, the idempotency bar, in and out of scope — then state their assumptions out loud and design against them. Asking ‘what’s the peak-to-average ratio’ and then actually using it at the whiteboard is the signal.
Game plan · how to rehearse
The principleActive recall beats re-reading. Don’t just read the answers — say them out loud, get graded, find your weak signals, and re-drill those. Six short days, ~30 minutes each.
Day 1 — learn the shape
Read the Walkthrough and System Map end to end — the mechanics, then where this sits in the larger system.
Run the Probe Drill in study mode: reveal every answer, don’t grade. You’re learning the 20 signals and the follow-up chains, not testing yet.
Day 2 — grade yourself
Run the Probe Drill in graded mode, cold. Mark each Solid or Revisit honestly — no half-credit.
Open Session progress: it lists the signals you flagged. Re-drill those until the answer comes without the prompt.
Day 3 — reconstruct & defend
Whiteboard recall: rebuild the whole design from cues alone, no peeking.
Trade-offs: for each fork, say the pick-when for both sides — never defend just one.
Model Answers: pick two or three angles and say them out loud, start to finish.
Day 4 — pressure
Mock run on the clock with Interviewer cuts in turned on — survive the interrupts.
Mixed fire: probes, curveballs, and trade-offs in random order — the register-switching is the hardest part. Retry the ones you fumble.
Day 5 — sharpen & track
Re-run Mixed fire and the weak Probe Drill signals from your session.
Save your Session code; paste yesterday’s to watch the trend climb.
Read the Cram sheet once — the spine, the decisions, the numbers.
Interview eve — 20 minutes
30-Second opener, out loud, twice.
One Mixed fire set — just to feel sharp.
Cram sheet, one pass. Then close the laptop and trust the reps.
