What a complete answer sounds like model script · the full arc, not just the opener
Frame it before diving"Before I trace it — I'll assume single region, files up to ~500 MB, and that downstream consumers need the result within a few seconds. Stop me if any of that's off."
Headline in one breath"At a high level: the object lands in S3, that fires a trigger, I hash and validate it while it streams, write the metadata transactionally, then fan it out to the processors — built so memory stays flat no matter the file size."
Walk the path"Concretely — the upload completes to S3 and emits an event. My handler opens a read stream and forks it through a PassThrough: one branch computes the SHA‑256, the other feeds the processor, so I never buffer the whole file or read it twice. Once I have the hash and size, I write the row — key, hash, status — to the metadata DB."
Name the risk yourself"The thing I'd flag without being asked is the dual‑write: the object is already in S3, but if the DB write fails I've got an orphan. The compensating delete covers the common case, but the real backstop is a reconciler that sweeps S3 for keys with no committed row. I wouldn't claim the delete alone is safe."
Name the trade-off line"For dispatch I'd run a Lambda per object at this volume — the simplest thing that works. The moment I need retries, ordering, or a DLQ, I'd put SQS in front and make the processors consumers. That's the line I'd move on, and I'd say so before they ask."
Be honest about the ceiling"This holds to low thousands of objects a minute. Past that the metadata DB is the bottleneck — I'd shard by tenant or key‑prefix and move fully onto SQS‑fronted consumers. I'd call that out as the next redesign rather than pretend it scales forever."
Interviewer: "What if two uploads race on the same key?"
Absorb the follow-up"S3 is last‑writer‑wins on the object, so the bytes are fine — but my metadata could interleave. I'd make the DB write idempotent on the key with a conditional put, so the second writer updates cleanly or no‑ops. And if ordering genuinely matters to the business, that's one more reason to move to a queue with a per‑key partition."
